Highlight #1 VCS

A VCS, otherwise known as a Versioning Control System, is kind of like a library for developers because they can check files out and check them back in. The only difference is that they can make changes to the files themselves. A VCS keeps track of who used what file last, when it occurred, and what the change actually was. Since it keeps track of every change, it allows users to even revert back to the original file if needed. Another great thing about VCS is that is a user is not supposed to have access, then the file that they want cannot be overwritten.

Highlight #2 Repositories

So what even is a repository? Repositories are kind of like a diary for web developers when using version control systems. Every time that a developer adds to or changes a VCS, the repository holds all the files added and it is easy to view its latest change. In a diary, it is easy to read about a person’s emotions and what they have done. Much like diaries, repositories track the history of a VCS and its status. Repositories can either be on a remote server and accessed by multiple users or on a local server and accessed by a individual.

Highlight #3 Git

To my understanding Git is a distributed versioning control system that is used on a more localized basis. So, it works in the same way as my definition for a VCS, but it differs on the number of people and who uses it. Git is used for different, individual projects so the members of the team working on the project can see progress faster. With less users and less distance to travel, processes are able to run smoothly and quickly. Git works in an efficient manner so many different projects can occur at the same time and there is no overlap.

Highlight #4 Git Commands

While working in Git, different commands are used to change files. Some basic commands that are use include ‘git add’, “git init’, ‘git commit’, and ‘git log’. The command ‘git add’ is used to take an almost quick picture of the work before it is committed to the history. The command ‘git init’ is used to start a new Git repository and is usually one of the first commands implemented. The next command I mentioned, ‘git commit’, is used to take whatever was previously added and save it to the history. And the final command I talked about, ‘git log’, lets someone look at previous changes.

Highlight #5 Collaborative Git

Git can be used individually if it is desired. But Git was actually created to be team-oriented. Git can be distributed so that multiple users have access to the same thing and can even work on a project simultaneously. When looking at tracking the history of collaborative Git, different tasks are being completed by different users, but can be worked on the same time. There is overlap among the time that users are making changes and once they choose to merge what they have been working on to the centralized copy of the Git they can see the overall progress.
